# Scene-Aware Split Fix — Technical Report (2026-07-04)

Closes finding 1 of `dataset_audit_2026-07-03.md` (train/val/test
near-duplicate leakage). Scope was deliberately narrow: fix the split
method, regenerate the dataset, verify, and document. No new features,
no other pipeline changes, D-Fire still pending (its raw folder is
empty and the merge step skips it as before).

---

## 1. Problem

The raw sources contain multiple Roboflow export variants of the same
original photograph, named `<original-stem>_jpg.rf.<32-hex>.jpg`. The
variants differ in bytes (augmented/recompressed), so the merge step's
md5 deduplication cannot see them, and the previous per-image split
scattered them across train, val, and test. The audit measured 16.9%
of val and 15.6% of test images with a near-duplicate in train,
inflating all evaluation metrics an estimated 5–10 mAP50 points.

## 2. Change (single file: `data_tools/split.py`)

- **`scene_key(file_name)`** — strips the `.rf.<32-hex>` suffix (regex
  `\.rf\.[0-9a-f]{32}$` on the stem); names without the suffix remain
  their own scene, so unaugmented sources (e.g. COCO) keep per-image
  behavior.
- **`assign_splits()`** — groups images into scenes, stratifies whole
  scenes by their *dominant* class signature (ties broken
  alphabetically for determinism), shuffles scenes with the fixed
  seed, and deals scenes whole to train → val → test until each
  split's image quota (70/20/10 of the group's images) is filled. A
  scene can never straddle two splits.
- **`splits.json`** now records `"method": "scene"` plus scene stats
  (total / multi-image / images covered / largest).
- **`split_report.md`** gained a "Leakage prevention" section.
- Public API, CLI, seed handling, and determinism guarantees are
  unchanged; `build.py` consumes `splits.json` exactly as before.

## 3. Tests

4 focused tests added to `tests/test_split.py` (16 total in the file):

- `scene_key` strips a real Roboflow suffix; plain and non-hex names
  are untouched.
- 30 synthetic scenes × 3 variants: every variant of a scene lands in
  the same split; scene stats correct.
- `splits.json` records `method: scene` and scene totals.

All 12 pre-existing split tests pass unmodified against the new
implementation. Full AI workspace suite: **239 passed** (was 235; +4
new). Backend/frontend untouched.

## 5. Verification results (independent script, all checks passed)

1. **Scene leakage: 0.** 9,956 scenes (1,061 multi-image covering
   3,650 images; largest 22 images); no scene appears in more than
   one split.
2. **Ratios:** 70.012% / 20.048% / 9.940% — within 0.1% of target.
3. **Class balance** (share of a split's images containing the class):
   fire 52.9/53.0/52.4%, smoke 46.7/46.6/46.7%, person 21.5/21.4/21.6%
   across train/val/test — max spread 0.56%. Boxes: train
   8,467/6,968/7,603, val 2,398/1,985/2,115, test 1,178/1,010/1,059
   (fire/smoke/person).
4. **Integrity:** every image has exactly one label twin in its split;
   no image assigned to two splits; processed file set exactly equals
   `splits.json` assignments; totals match (12,545).
5. **Residual near-duplicate overlap** (exact 64-bit dHash vs train,
   the audit's method): val **8.0%** (was 16.9%), test **6.2%** (was
   15.6%). Overall duplicate clusters essentially unchanged
   (1,049 clusters / 2,690 images / 21.4% vs the audit's
   1,038 / 2,617 / 20.9% — method agreement), confirming the fix
   moved duplicates into the same split rather than removing them.

## 6. Residual risk and follow-ups (not in scope here)

- The remaining ~8%/6% overlap comes from visually identical images
  whose file names share no Roboflow stem (adjacent video frames,
  re-uploaded copies). Options, in the audit's recommendation 7:
  prune duplicate clusters, or group scenes by dHash cluster instead
  of file name. D-Fire (partly video-derived) should be merged before
  deciding, since it will add its own clusters.
- One of the two negative images travels inside a fire/smoke-dominant
  scene and therefore sits in test; the signature table in
  `split_report.md` counts scenes by dominant signature, so it shows
  1 negative — this is expected scene-aware behavior, and the
  validator still reports both negatives.
- **Phase 8D metrics (mAP50 0.509) were measured on the old split and
  are not comparable to anything evaluated on the new split.** The
  upcoming 50-epoch run must be trained and evaluated on this split
  only.
